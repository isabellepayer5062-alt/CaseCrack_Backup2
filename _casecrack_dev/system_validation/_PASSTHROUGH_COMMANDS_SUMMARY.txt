# Passthrough Commands: What Actually Matters

## The Challenge
With 50+ passthrough commands, random hardening wastes effort. **Need**: Data-driven prioritization.

## The Solution

- **Volume** - which commands dominate traffic? (80/20 analysis)
- **Tenant Count** - which have broad adoption? (impact assessment)
- **Error Rate** - which are fragile? (risk assessment)

## Red Flags to Watch

⚠️ **Error rate spike on core commands**
- May indicate recent code changes or resource issues
- Investigate immediately; may block rollout

⚠️ **Single-tenant commands in Phase 1**
- Unusual; may indicate incomplete metric collection
- Verify tenant_id is being set correctly

⚠️ **Uniform error rates across all commands**
- May indicate outcome recording is broken
- Check if `record_request()` is being called for all passthrough invocations

⚠️ **No multi-tenant commands**
- Either using single tenant in demo, or metric collection needs fixing
- Verify: are passthrough calls distributed across multiple tenants?

## FAQ

**Q: Why only Phase 1 first?**
A: 76% concentration means you get 76% of the benefit from 3 commands. Start there, prove success, then move to Phase 2.

**Q: What if my distribution is different?**
A: The analyzer is flexible. Categories auto-adjust based on your data. Run it to see.

**Q: Should I harden Phase 2 while doing Phase 1?**
A: Probably not. Finish Phase 1, validate it works, then audit Phase 2 to understand risks. Sequential is safer.

**Q: What about Phase 3?**
A: Defer. If business need arises, recategorize and plan then. Currently <5% benefit.

**Q: How do I know the analysis is correct?**
A: Compare against your support tickets / incident logs. Do high-error commands match known issues?

**Q: Can I run this analysis continuously?**
A: Yes! Set up hourly/daily cron job. Track trends. Alert on error rate spikes.
